## §1 "release 版翻墙服务全 timeout" —— 实测证伪,根因是旧包/已修竞态

**现象主诉**:release 版下 claude.ai/gmail 全 timeout、无 error log;debug 版正常。

**排查(3 个 agent 反编译 release DEX + 真机实测)**:
- mapping/DEX 证明:`RuleAction` 未被 enum-unbox、三个单例完好,`decide()` 五分支与 `connect()` 的
  `if(bypassVpn)` 守卫完好 → **「R8 把 PROXY 误判成 DIRECT」缺乏证据**。
- 真机(含 race fix `e573824` 的新构建)实测:claude.ai 判定 `-> PROXY`、经 VPN `200`、
  8 墙外域名 ×3 并发 **24/24 全 OK**、完整 HTTP relay 往返正常 → **复现不出 timeout**。
- 图标佐证:debug 与 release APK 的 launcher 前景矢量 `pathData` **逐字节相同** → release 是最新代码、非缓存。

**结论**:用户当时遇到的极可能是**更早手动构建的旧 release 包**(代码旧、缺 `e573824` 广告表快速开关竞态修复),
"release 坏 / debug 好" 是「旧包 vs 新代码」之差,**不是 R8 优化 bug**。诊断期临时加的 `-dontoptimize` 已回退,
保持标准 optimize 配置。

## §2 访问慢 —— 瓶颈是 VPN 出口 RTT,代理与并行度都清白

Mac Safari 经手机代理访问 claude.ai 慢。逐层实测(与 Safari 同路径):
- `ping` 纯往返:VPN 隧道 `8.8.8.8` = **203ms**、claude.ai(160.79.104.10)= **207ms** → RTT 物理 200ms。
- 经代理 ≈ 手机直连(都 ~460ms TCP)→ **代理转发开销 <10ms**,不是瓶颈。
- DNS 在 Mac 端 1ms;代理端有 `dnsCache`,非瓶颈。
- **并行度有效**:10 并发 claude.ai 总 **0.84s** ≈ 单个 0.98s(串行需 7.88s)→ relay 池零排队;调高 preset 无益。
- **连接复用是关键**:经代理复用连接,第 2/3 个请求 **0.22~0.30s**(1 个 RTT),全新握手 0.94s。
  手机浏览器复用 + HTTP/3(`alt-svc: h3`,代理转不了 QUIC)→ 体感更快。

**结论**:慢 = ①首次全新握手(多个 200ms RTT)②VPN 隧道 200ms RTT ③代理转不了 QUIC。
代理侧已优化到位(IPv4 优先 / dnsCache / Happy Eyeballs / TCP_NODELAY / 独立 DNS 池),无可挖。
唯一杠杆:换低延迟翻墙节点。

## §3 错误日志降噪 + bind 重试 + 端口占用提示(1.1.2)

**导出日志分析**:244 行里 **1 个真问题 + 59 条噪音**:
- 真问题(1 次):`SOCKS5 bind :1080 failed (EADDRINUSE)` —— SO_REUSEADDR 救不了仍在 LISTEN 的端口,
  根因是 stop→start 快速重启时旧 socket 未释放(或别的 app 占用)。
- 噪音(59 条,全来自单个客户端 192.168.50.148):`JobCancellationException`×53 / `Connection reset`×4 /
  `Broken pipe`×2 —— 浏览器正常断开,被无差别记进 FileLog,淹没真问题。

**改动**(`core/proxy/ProxyServer.kt`):
- 新增 `isPeerClosed(e)`:识别连接重置/管道断开/socket 关闭/协程取消 → 走 `Log.d`、**不写 FileLog**;
  仅真正的服务端错误进 `FileLog.w`。
- bind 失败对端口占用**重试 3 次(间隔 150ms)**(`BIND_RETRY_ATTEMPTS`/`BIND_RETRY_DELAY_MS`),
  缓解 stop→start 端口释放竞态。

**端口占用提示**(`ui/dashboard/DashboardScreen.kt` + `values(-zh)/strings.xml`):
- 主页新增 `errorContainer` 告警卡,`portBindErrors` 非空时显示「端口被占用，对应代理未启动：SOCKS5 :1080…」
  + 引导去设置改端口(`port_bind_failed_banner`,中英双语)。此前仅 Settings 端口框标红,主页启动看不到。

## §4 版本号规则(已落记忆 [[bump-version-every-build]])

每次构建前递增三段式 `MAJOR.MINOR.PATCH`(修复 +PATCH、功能 +MINOR、重大 +MAJOR),`versionCode` 单调 +1。
**动机**:版本长期停在 `1.0` 时真机无法区分「装的是哪一次构建」,正是 §1 误判旧包的根源。装机后系统「应用信息」页可见版本。

## §5 键盘遮挡底部输入框修复(1.1.3)

**根因**:`MainActivity` 用 `enableEdgeToEdge()`,内容延伸到键盘区域下;而全项目 **无任何 `imePadding`/`bringIntoView`**,
仅靠 `AppRoot` 的 `Scaffold(contentWindowInsets = safeDrawing)` 单点兜底。在 `verticalScroll` 长页面 + 底部输入框
(端口 / 域名)场景,焦点框没被滚到键盘上方 → 被软键盘遮住、看不清输入。

**改动**:
- `ui/AppRoot.kt`:`contentWindowInsets = WindowInsets.safeDrawing.exclude(WindowInsets.ime)` —— 系统栏/刘海仍由
  Scaffold 处理,**IME 交给各内容页**(否则 `consumeWindowInsets` 会吃掉 IME,页内 `imePadding` 失效)。
- 4 个含输入页的滚动容器加 `Modifier.imePadding()`(在 `verticalScroll` 外层):`SettingsScreen` / `RulesScreen` /
  `RuleSetEditScreen`(Column)、`RuleSetManagerScreen`(LazyColumn)。键盘弹出时滚动视口收缩,Compose 自动把聚焦框滚到键盘上方。

**验证**:编译通过、装机 1.1.3、app resumed 无崩溃。⚠️ Pixel Fold 折叠屏 + 无线 adb 截图不可用(双 display 截到熄屏),
键盘交互的视觉确认留待真机手测;改动为 Compose 官方标准做法,风险低。

## §6 SplashScreen API(1.1.4)

**目标**:消除冷启动「图标 → 空白 → 主界面」的跳变(`MainActivity` 在 `showOnboarding==null`、DataStore 首读期间画空白)。

**改动**:
- 依赖:`androidx.core:core-splashscreen 1.0.1`。
- `res/values/themes.xml`:`Theme.HxmyProxy.Starting`(parent `Theme.SplashScreen`)——
  `windowSplashScreenBackground`=薄荷绿(`@color/splash_background`,沿用 launcher 背景起色 `#A7F3D0`)、
  `windowSplashScreenAnimatedIcon`=`@drawable/ic_launcher_foreground`(同 launcher 前景矢量)、
  `postSplashScreenTheme`=`@style/Theme.HxmyProxy`。
- `AndroidManifest.xml`:`MainActivity` theme → `Theme.HxmyProxy.Starting`。
- `MainActivity.kt`:`installSplashScreen()`(在 `super.onCreate` 前)+ `setKeepOnScreenCondition { !contentReady }`;
  `setContent` 内 `showOnboarding != null` 时置 `contentReady=true` → **启动屏保持到首屏数据就绪才退场**,衔接无空白。

**验证**:编译通过、装机 1.1.4、冷启动 ×3 `TotalTime` 372~506ms(与接入前 433~447 相当,keepOnScreen 仅在数据未就绪时短暂保持)、
无崩溃、系统 log 显示 `splashScreenStyle` 机制激活。视觉(薄荷绿+图标)受 Pixel Fold 折叠屏截图限制未截图,主题/API 正确。

## §7 Baseline Profile(1.1.5)

**目标**:release 首启 AOT 预编译应用热路径,降低冷启动耗时与首屏掉帧(用户反馈「图标加载完进主界面有卡顿」;
§见启动链路排查已排除主线程同步 IO,根因是无 baseline profile → 启动/Compose 代码解释执行+JIT)。

**改动**:
- 依赖:`androidx.profileinstaller 1.4.1`(首启把 baseline profile 安装进 ART;由 AndroidX App Startup 自动初始化,无业务代码)。
- `app/src/main/baseline-prof.txt`(手写基础版):通配覆盖本应用全包
  `HSPLcom/mzstd/hxmyproxy/**->**(**)**` + `Lcom/mzstd/hxmyproxy/**;`。
  AGP 的 `expandReleaseArtProfileWildcards` 展开通配 → `compileReleaseArtProfile` 编进 `baseline.prof`。
- Compose / AndroidX 框架的启动热路径由各依赖库自带 baseline profile 覆盖,本文件只管 app 自身包。

**验证**(Pixel Fold 真机):
- APK 含 `assets/dexopt/baseline.prof`(7166B)+ `baseline.profm` + profileinstaller 版本标记。
- 广播 `ProfileInstallReceiver` 安装 → `result=1`;`cmd package compile -m speed-profile` 后 dexopt `[status=speed-profile]`(arm64 主 abi)。
- **冷启动 `TotalTime`:接入前 433/440/447ms → AOT 后 296/289/329/297ms,降约 31%**。
- 注:Play 安装会自动应用 baseline profile(Cloud profile);本地 `adb install` 需首启或广播触发 ProfileInstaller 后下次启动生效。

**可升级**:后续接 `androidx.baselineprofile` 插件 + `:macrobenchmark` 测试模块,在真机录制实际调用的精准热路径 profile
替换手写通配版(更省 APK/dexopt 体积、更贴合真实启动轨迹)。
