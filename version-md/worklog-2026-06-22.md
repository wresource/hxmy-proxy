## 4.5 交付后用户驱动的改进（同日续）

## 4.10 Compose UI 测试落地（本次会话）

目标：完成测试项 #2（Compose UI 自动化）。最终 `MainUiTest` 3 测试全绿；**所有测试：JVM 单测 40/40 + 仪器连接 5/5**。过程中逐个排障，连带修了 2 个真实 bug 和 2 处基础设施问题——记录如下，便于以后复用经验。

**真实 bug #1 — 切语言把用户弹回主页（`ProvideAppLocale` 重挂载）。**
原 `ProvideAppLocale` 有结构性分支：`SYSTEM`（locale==null）时**直接** `content()` 并 `return`，非 SYSTEM 时把 `content()` 包进 `CompositionLocalProvider`。从「跟随系统(英文)」切到「中文」会**跨越 null 边界**，`content()` 的调用点在 slot table 中位置改变 → 整个子树（含 `rememberNavController`/`NavHost`）被销毁重建 → 导航重置到起始页 `Dashboard`。表现：用户在设置页改语言，被莫名弹回主页。
是 UI 测试 dump 语义树才发现的——超时时树已是中文（语言切换其实成功了）但 `Selected=true` 的 tab 是「主页」，停在了 Dashboard。
修复（`AppLocale.kt`）：**结构恒定**，无论是否 SYSTEM 都走同一个 `CompositionLocalProvider`，`null` 时提供原始 config/context（no-op）。结构不变 → 切语言不再重挂载 → 留在当前页。

**真实 bug #2 — `android.util.Log` 未 mock，导致 `ProxyIntegrationTest` 全部超时挂死。**
为「连接日志」功能，`ProxyServer` accept 循环每次 accept 都调 `Log.i(...)`（`HttpProxyServer`/`Socks5ProxyServer` 也有）。JVM 单测里 `android.util.Log` 默认**抛异常**（"not mocked"），连接处理协程在写响应前就崩了 → 客户端读永远阻塞。真机 `Log` 正常，故设备上没问题、只有 JVM 测试中暴露。
修复（官方「本地单元测试」推荐）：`build.gradle.kts` 加 `testOptions { unitTests { isReturnDefaultValues = true } }` → 未 mock 的 `android.*` 返回默认值而非抛异常。修后 `ProxyIntegrationTest` 8/8 通过、0.19s（之前 64s 全超时）。
连带健壮性：给 `ProxyIntegrationTest` 客户端 socket 统一加 `SO_TIMEOUT=8000`（新 `client(port)` 助手）。否则任一阻塞 read 会**永久挂死整个测试 JVM**（JUnit `@Test(timeout)` 靠 `Thread.interrupt` 无法中断阻塞 socket 读）——卡住改为抛 `SocketTimeoutException` 快速失败。正是靠这个超时才把「无响应」从无限挂起变成可观测的失败，进而定位到 bug #2。

**基础设施 #1 — Espresso 3.5.1 与 API 37 不兼容。**
Compose 测试的空闲同步底层走 Espresso，旧 Espresso 用反射调 `InputManager.getInstance()`（API 37 已移除）→ `NoSuchMethodException`，3 个测试 ~10ms 即崩。
修复（`libs.versions.toml`）：`espressoCore 3.5.1→3.7.0`、`ext-junit 1.1.5→1.3.0`（官方 Maven 查得的最新稳定版）。

**基础设施 #2 — 全局 Gradle HTTPS 代理主机配错。**
`~/.gradle/gradle.properties` 里 `systemProp.https.proxyHost=192.168.50.65`（不可达），而 `http.proxyHost` 和 shell env 的 `HTTPS_PROXY` 都是 `192.168.1.103`。之前能编译是因依赖都在缓存；espresso 升级拉新依赖才触发 HTTPS 下载超时。
修复：改 `https.proxyHost` 为 `192.168.1.103`（与 http/env 一致）。**注意：这是仓库外的全局文件，不在本次提交内**；该文件带今天时间戳、疑似 Android Studio 自动写入，若 IDE 再次写回错值需重修。

**测试写法经验（`MainUiTest`）：**
- 导航**用 `testTag`（`nav_<route>`）**，不要用字形/本地化文案：`NavigationBarItem` 合并语义后默认（merged）树匹配不到图标 Text（框架会提示 `useUnmergedTree`）。在 `AppRoot` 给每个 item 加 `Modifier.testTag("nav_$route")`。
- 语言切换是**异步链路**（点击→ViewModel→DataStore 写→Flow→uiState→重组），不在 Compose idle 跟踪内。直接断言会竞态（时而找到时而找不到）。用**手动轮询**（`waitForIdle` + sleep）等本地化文案出现；超时则 `onRoot().printToLog(...)` dump 语义树到 logcat 诊断（就是这一步发现 bug #1）。
- 断言用「存在性」而非 `assertIsDisplayed`：标题是滚动列表顶端 SectionTitle，像素可见性判定不稳；`"语言"`/`"Language"` 各自只在对应语言下存在，存在即证明双向切换成功。

**结果：** `:app:testDebugUnitTest` 40/40 + `:app:connectedDebugAndroidTest` 5/5（MainUiTest 3 + SettingsRepositoryInstrumentedTest 2）全绿。

## 4.13 修三个问题：图片/小文件(keep-alive) + 在线连接数 + 总流量统计（本次会话）

用户反馈：①图片/小文件加载不行；②连接数应是在线而非累计；③需要统计本次会话总流量。先用并行 workflow（3 agent）做根因调查，再实现+测试。

**① 图片/小文件 = 缺 HTTP keep-alive（真功能 bug，已修）。**
根因：普通 HTTP 转发路径**一个客户端 TCP 连接只处理一个请求**就关闭——`handle()` 不循环、对上游强制 `Connection: close`、relay 在上游 EOF 后 `shutdownOutput` 客户端写端。浏览器加载网页会**复用同一连接**发几十个小图/资源请求，首个之后全丢 → 图片/小文件加载失败；大文件(单请求单连接)恰好正常。（排除误区：relay 二进制安全，非解码问题。）
修复：重写 `HttpProxyServer.handle()` 为 **keep-alive 请求循环**——新建 `HttpForwarding.kt` 做 RFC 7230 报文定界（Content-Length / chunked / 读到关闭），逐请求新建上游(对上游 close 取干净定界)但**保持客户端连接复用**；按响应定界决定是否保活（不确定定界则对客户端 close）；新增请求间空闲超时 `KEEPALIVE_IDLE_MS`。CONNECT 盲隧道路径不变。普通 HTTP 不再走 RelayEngine，故流量计量改由共享 `trafficSink` 注入 `HttpProxyServer`。
复现/回归测试：`ProxyIntegrationTest.httpKeepAliveServesMultipleBinaryRequests`——同一客户端 socket 连发两个请求、取回两份**二进制**响应逐字节校验（旧代码第二请求 EOF/超时）；`HttpForwardingTest` 单测覆盖定界决策 + chunked/length 二进制安全拷贝。

**② 连接数 = 已是在线表，但停止/重启不清零会"卡高"（已修）。**
调查发现代码本就是在线实时表（`ConnectionRegistry` accept +1 / close −1），并无累计计数器。但 `registry` 是 `@Singleton` 且 **stop/热重启从不清零**——停止后在途 relay 要等空闲超时才 `release()`，残留计数带到下次会话，表现为"像累计/不回落"，很可能正是用户所见。
修复：`ConnectionRegistry` 加 `reset()`（stop/start 调用清零）+ `onChange` 回调（accept/close/reset 即时推送在线数到 UI，不必等 1s ticker）。单测覆盖 reset 归零、onChange 实时值、被拒不触发。

**③ 本次会话总流量（已加）。**
`totalUp/totalDown` 本就累计（供速率），但没暴露给 UI、Start 时也没清零。新增 `ShareState.totalBytes`，Start 清零、1s ticker 发布 `up+down`；`Format.formatBytes`（B/KB/MB/GB）+ Dashboard 新增一行 + 中英文 `total_traffic` 字符串。`FormatTest` 单测。

**验证：JVM 单测 50/50（40→50）+ 真机 ProxyLoadTest 9/9 全绿；吞吐 148 MB/s、50 并发零失败、外网可达——keep-alive 重写无回归。**

## 4.18 出站连接 Happy Eyeballs（RFC 8305）——降低 Cloudflare/Stripe 类站点尾延迟（本次会话）

**背景（用户诊断）。** 用户反馈访问 Stripe/Cloudflare 类站点经代理特别慢。核对代码定性：代理本身不加延迟（上下游 `tcpNoDelay`、relay 即时 flush、DNS 走 VPN），慢主要来自**出口 IP 信誉触发风控挑战 + 这类站点多域名串行握手 × VPN 出口 RTT**——多属出口/站点策略。代理这侧唯一能改的是 `OutboundConnector` 的连接策略（另一项 QUIC 回退用户暂不做）。

**问题：顺序回退会干等满超时。** 原 `connectAny` IPv4 优先后**逐个**尝试，每地址 5s 超时；多 anycast 站点首个 IP 在 VPN 路径上慢/不可达时，要等满 5s 才换下一个。

**改为 Happy Eyeballs（RFC 8305）交错并行。** 起第一个地址，**每 250ms（`HE_ATTEMPT_DELAY_MS`，RFC 默认）或当前尝试失败**就并行起下一个，首个成功者胜出、其余在途连接立即关闭（关 socket 中止其阻塞 `connect`）。各地址 5s 超时**并行**故总耗时≈最快可达者，不再累加。`connect()` 改 `suspend`（调用方 SOCKS5/HTTP `handle` 本就是 suspend）。新单测 `racesPastUnreachableFirstToReachableSecond`：首地址 TEST-NET-1（黑洞）→ ~0.26s 连上次地址 echo，远早于 5s，证明无干等。

**对抗式审查（13 agent / 3 lens）+ 修复。** 0 critical/high，4 条确认全修：
- 🟡 `withTimeoutOrNull(receive())` 取消 `receive` 时可能**丢弃已投递的结果**（Channel.receive 非取消原子）→ `pending` 虚高、永久挂起。修：改用 `select { onReceive; onTimeout }`（互斥子句，不丢元素）。
- 🟡/🟡 取消时"已建 socket 但晚于 finally 注册"的落单尝试逃过清理 → 可能 5s 阻塞 + FD 泄漏。修：加 `closed` 闸门（`AtomicBoolean`，与注册同锁），finally 也置位，晚到者见 `closed` 自行关闭。
- ⚪ `while(true)` 后不可达 `throw` 死代码。修：改显式终止条件 `while(pending>0 || nextIdx<size)`，退出即抛（throw 变可达）。

**验证：JVM 56/56（+1 race 测试）+ 仪器 21/21 全绿，无回归。**
