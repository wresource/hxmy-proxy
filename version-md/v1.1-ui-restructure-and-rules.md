## 3. Phase 2 — 规则引擎（分流 + 广告拦截）

### 3.1 架构 · 两个落点
hxmy 是代理:客户端把请求发来后只能**转发**或**拒绝**,**无法命令客户端「直连」**。故:
- **「直连/绕过代理」只对用 PAC 的客户端有效** —— PAC 对直连名单域名返回 `DIRECT`,客户端自己直连、不经手机。手动填代理的客户端所有流量都过 hxmy(没法绕,但仍走手机出口)。
- **「阻断广告/REJECT」对所有客户端有效** —— hxmy 代理层拒绝这些域名的连接。

→ 规则引擎 = **落点① PacGenerator 注入(DIRECT/PROXY)** + **落点② 代理层 REJECT**。

### 3.2 数据模型（新 `core/rules/` 包）
- `RuleEngine`:持**后缀字典树**(域名按 `.` 切分、标签反转插入,匹配 O(标签数));`decide(host) → DIRECT / PROXY / REJECT`。
- **来源三层 + 优先级**:`用户自定义 > App 专项 > 大类(direct/proxy) > IP 兜底`;广告表并入 REJECT 层,**用户白名单覆盖优先级最高**(防误杀)。
- **PSL 边界**:合并/去重时剔除「只到公共后缀」(`co.uk`/`github.io`)的危险项,别拦整段。
- 存储:DataStore 存各组开关 / 用户规则 / 订阅 URL;内置预处理清单进 `assets`;订阅后台拉取 + 本地缓存 + 失败回退内置。

### 3.3 落点① PAC `core/proxy/PacGenerator.kt`
`generate(...)`(`:17`)加规则参数:对 host —— 直连名单→`DIRECT`、广告→黑洞/`DIRECT`(配代理层 REJECT)、其余→`PROXY <入口>`。

### 3.4 落点② 代理层 REJECT
在拿到目标 host 处调 `RuleEngine.decide(host)`,`REJECT`→拒绝(关连接 / 502):
- `HttpProxyServer.handleConnect`(CONNECT host)与 `forwardPlainHttp`(`uri.host`);
- `Socks5ProxyServer`(ATYP 解析出的 host/addr)。
- 现有 `EgressGuard.isAllowed(addr)` 是 `OutboundConnector.connectAny` 里唯一的出口过滤点,可作 IP 级 REJECT 的参考/补充挂点。

### 3.5 规则页 `ui/rules/RulesScreen.kt`（三模块,分级开关 — 澄清 C）
1. **IP / 域名名单**:整组一个开关 + 增删改(走 DIRECT)。
2. **App / 服务规则集**:每个服务一个开关(网易云/微信/Telegram/Spotify…)+ 自定义订阅。
3. **广告拦截**:每个表一个开关(默认 OISD small、**默认关**)+ 用户白名单覆盖。

## 5. 关键技术注意

- PAC 直连只对 **PAC 客户端**绕过;手动代理客户端仍过 hxmy(走手机出口)。广告 REJECT 对**所有**客户端有效。
- License:内置只用 **GPL-3 / MIT**;**SukkaW(AGPL)仅作订阅 URL、不打进包**;Peter Lowe 未明确授权再分发,默认不内置。App 内标注来源 + License。
- 体积:small/light 量级明文 ~1–3 MB、gzip ~200–500 KB、内存几 MB,可内置;big/pro 量级走订阅。
- **VpnDownStrategy 与规则正交**:`settings_vpn_strategy` 三态(继续走普通网络 fail-open / 阻断 fail-closed 默认 / 警告但继续)已实现,技术可行(出口跟随系统默认网,D4);它管「VPN 断了之后**整体**继续/停」,规则管「**每条**流量 direct/proxy/reject」,可共存。「继续走普通网络」是安全取舍(VPN 掉→别人流量从真实 IP 出,故默认阻断)。
